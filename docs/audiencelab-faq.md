#### Understanding Lists, Segments, and Tags

AudienceSend gives you three powerful tools to manage and organize your contacts: **Lists**, **Segments**, and **Tags**. Each plays a different role in how you group, filter, and target your audience.

A **List** is a static group of contacts you upload or sync into AudienceSend.

- Think of a list as a spreadsheet or file of people you‚Äôre uploading together

- Lists don‚Äôt change unless you manually add/remove contacts

- Great for cold uploads, ad exports, or lead magnet opt-ins

- Need to segment audiences by how they were acquired

A **Segment** is a saved filter that dynamically groups contacts based on behavior, metadata, or status.

- You create them by applying filters like: ‚ÄúHas opened an email in last 7 days,‚Äù ‚ÄúWas tagged `warm_lead`,‚Äù etc.

- Segments are perfect for re-engagement or automation targeting

A **Tag** is a simple label you attach to a contact (or group of contacts) to track their status, source, or behavior.

- Tags are the most flexible tool for labeling contacts

- Mark contacts for inclusion/exclusion in future campaigns

- Trigger follow-ups based on actions (e.g., ‚Äútag if replied‚Äù)

### üîÅ Syncs Overview ‚Äì Understanding Data Syncs in AudienceSync

**Syncs** define how your data is **transferred and mapped** from its source to its destination within **AudienceSync**.

- A **destination** (where the data should be delivered).

AudienceSync supports **two sync types**, each serving different use cases:

- **What it does:** Replaces _all_ the data in the destination with the data from the source.

- **When to use it:** When you want to completely overwrite the destination with the latest version from your source.

- **What it does:** Syncs only the data that has **changed since the last sync**.

- **When to use it:** Ideal for keeping your destination up to date with **only new or updated records** without overwriting existing data.

In **AudienceSync**, **streams** refer to the **destination APIs or objects** you are syncing data to.

Each stream represents a different target object in the destination platform where specific data types are synced.

<a id="audiencesync-cdp-incremental-syncs-with-cursor-field-minimizing-data-transfer-in"></a>

**Source:** https://docs.audiencelab.io/audiencesync-cdp/4z72nzTBAcUz3cT3wdduZB/cursor-fields/4z72nzTBAcqqMd2WDFBcc4

### üîÅ Incremental Syncs with Cursor Field ‚Äì Minimizing Data Transfer in AudienceSync

AudienceSync supports an optimized **Incremental Sync** mode using a **Cursor Field**. This allows you to transfer **only new or updated records** by leveraging a field that tracks changes over time‚Äîresulting in **faster syncs** and **less data duplication**.

> üîç This feature is only available for sources that support cursor fields (currently only date fields are supported).

A **Cursor Field** acts as a **timestamp marker** to track what data has changed since the last sync.

- Ensures only **modified or added records** are synced.

- Enables the system to **resume exactly where it left off**.

- Optimizes performance and reduces API strain or data volume.

> ‚ö†Ô∏è Do not edit or remove a cursor field after setup‚Äîit may cause data loss or disrupt sync integrity.

|     |     |     |
| --- | --- | --- |
| Name | Plan | Updated At |
| Charles Beaumont | free | 2024-04-20 10:00:00 |
| Eleanor Villiers | free | 2024-04-20 11:00:00 |

```sql
SELECT * FROM source_table
WHERE updated_at >= '2024-04-20 10:00:00';
```

‚û°Ô∏è Both records are synced (all are considered new).

|     |     |     |
| --- | --- | --- |
| Name | Plan | Updated At |
| Charles Beaumont | free | 2024-04-20 10:00:00 |
| Eleanor Villiers | paid | 2024-04-21 10:00:00 |

```sql

WHERE updated_at >= '2024-04-20 11:00:00';
```

|     |     |     |
| --- | --- | --- |
| Name | Plan | Updated At |
| Charles Beaumont | paid | 2024-04-22 08:00:00 |
| Eleanor Villiers | pro | 2024-04-22 09:00:00 |

```sql
SELECT * FROM source_table
WHERE updated_at >= '2024-04-21 10:00:00';
```

#### ‚öñÔ∏è Handling Ambiguity & Inclusive Cursors

If the cursor field is too coarse (e.g., tracks **only dates**), distinguishing between **old and new records** within the same day may be difficult.

We implement **"at least once delivery"**‚Äîso even if there's ambiguity, data is **safely re-sent**. It avoids missing updates but may result in duplicates, which should be handled downstream.

- If a record changes **without updating the cursor field**, it **will not be picked up** in incremental syncs.

- **Editing or deleting** a cursor field **breaks the tracking system**, leading to **potential data loss or inconsistencies**.

> üß∑ To maintain sync reliability: Never change or remove the cursor field after configuration.

### üîÅ Full Refresh Syncs ‚Äì How They Work in AudienceSync

**Full Refresh** is a sync mode in **AudienceSync** used when you want to **completely replace existing data** in the destination with new data from the source.

This method retrieves **all available data**, regardless of what was previously synced, ensuring the destination always holds a **full, current snapshot** of the source.

- You want to **overwrite** all data in the destination.

- You need a **clean slate** with the latest source data.

- It‚Äôs okay to **lose old records** that are no longer present in the source.

- The system **deletes existing rows** in the destination table.

- It then inserts the **entire dataset** from the source.

Let‚Äôs say the destination contains a `Users` table:

|     |     |     |
| --- | --- | --- |
| id | name | email |
| 1 | Alice | alice@example.com |
| 2 | Bob | bob@example.com |

|     |     |     |
| --- | --- | --- |
| id | name | email |
| 1 | Alice | alice@example.com |
| 3 | Carol | carol@example.com |
| 4 | Dave | dave@example.com |

> üìù Notice: Bob's record was removed, and new users Carol and Dave were added.

##### ‚úÖ Step 1: Prepare a Webhook Testing URL

##### ‚úÖ Step 2: Create a New HTTP Destination in AudienceSync

1. Navigate to `Destinations ‚Üí Add Destination ‚Üí HTTP`

For each header, map the **Key** and **Value**:

`Content-Type` ‚Üí `application/json`

> üîê You must explicitly map each header field in the destination setup screen. This ensures that the headers are included correctly in every outgoing request to the webhook or API endpoint.

In AudienceSync's destination setup screen, you'll see a section labeled **"Headers"**. Here, you must manually define each header by specifying a key-value pair. These headers are sent along with the HTTP request and are often required by APIs for proper request handling, authentication, or even routing.

##### üîß Examples of Header Mappings (Including User Data Fields)

Sometimes you'll see custom headers that begin with `x-`, like `x-user-name`. The `x-` prefix is a conventional way to indicate that a header is non-standard or proprietary ‚Äî in other words, it's not part of the official HTTP specification like `Content-Type` or `Authorization`.

You **do not need** to use the `x-` prefix unless the destination API specifically requires it. It's just a naming convention. If you're sending metadata like a user's name, email, or phone number, and the receiving endpoint expects plain field names, you can name the headers without the `x-` prefix ‚Äî such as `name`, `email`, or `phone_number`.

|     |     |     |
| --- | --- | --- |
| Header Key | Header Value | Purpose |
| `Content-Type` | `application/json` | Tells the server the format of the request body |
| `Authorization` | `Bearer your_token_here` | Required for Bearer token authentication |
| `name` | `John Doe` | Custom metadata: user's name passed in header |

### üîÅ Incremental Syncs ‚Äì Syncing Only New or Updated Data in AudienceSync

**Incremental syncs** transfer only the data that has **changed since the last sync**‚Äîthis includes **new or updated records**, minimizing data transfer and reducing redundancy.

This method avoids resending previously synced data by using a **unique primary key** and a reliable update-tracking field (like `updated_at`).

> üìù Note: The first incremental sync behaves like a full refresh because all data is considered new.

1. A **unique primary key** (set during model creation) is used to identify each row.

2. AudienceSync checks for changes using a column like `updated_at`, `modified`, or `timestamp`.

3. Only **new or modified rows** are included in each sync.

4. Destination data is **updated or appended** as needed.

  FROM all_sources

```sql
SELECT
  -- Combined Email Field
  COALESCE(
    NULLIF(TRIM(split_sha256[idx]), ''),
    NULLIF(TRIM(split_sha256_business_email[idx]), '')
  ) AS normal_email,

-- Combined Phone Field
  COALESCE(
    NULLIF(TRIM(split_mobile_phone[idx]), ''),
    NULLIF(TRIM(split_personal_phone[idx]), ''),
    NULLIF(TRIM(split_direct_number[idx]), '')
  ) AS normal_phone,

-- Static fields (not exploded)
  FIRST_NAME,
  LAST_NAME,
  PERSONAL_CITY,
  PERSONAL_STATE,
  PERSONAL_ZIP,
  GENDER
FROM (
  SELECT
    -- Splitting multi-value fields
    split(SHA256_PERSONAL_EMAIL, ',') AS split_sha256,
    split(PERSONAL_EMAILS, ',') AS split_emails,
    split(DIRECT_NUMBER, ',') AS split_direct_number,
    split(MOBILE_PHONE, ',') AS split_mobile_phone,
    split(PERSONAL_PHONE, ',') AS split_personal_phone,
    split(SHA256_BUSINESS_EMAIL, ',') AS split_sha256_business_email,

-- Static fields passed through
    FIRST_NAME,
    LAST_NAME,
    PERSONAL_CITY,
    PERSONAL_STATE,
    PERSONAL_ZIP,
    GENDER,

-- Index list for row explosion
    range(
      0,
      GREATEST(
        array_length(split(SHA256_PERSONAL_EMAIL, ',')),
        array_length(split(PERSONAL_EMAILS, ',')),
        array_length(split(DIRECT_NUMBER, ',')),
        array_length(split(MOBILE_PHONE, ',')),
        array_length(split(PERSONAL_PHONE, ',')),
        array_length(split(SHA256_BUSINESS_EMAIL, ','))
      )
    ) AS idx_list
  FROM all_sources
) t,
UNNEST(idx_list) AS idx_struct(idx)
WHERE (
  (split_sha256[idx] IS NOT NULL AND split_sha256[idx] <> '')
  OR (split_sha256_business_email[idx] IS NOT NULL AND split_sha256_business_email[idx] <> '')
)
AND (
  (split_mobile_phone[idx] IS NOT NULL AND split_mobile_phone[idx] <> '')
  OR (split_personal_phone[idx] IS NOT NULL AND split_personal_phone[idx] <> '')
  OR (split_direct_number[idx] IS NOT NULL AND split_direct_number[idx] <> '')
);
```

- Combines **SHA256\_PERSONAL\_EMAIL** and **SHA256\_BUSINESS\_EMAIL** into a single column called `normal_email`

- Merges **MOBILE\_PHONE**, **PERSONAL\_PHONE**, and **DIRECT\_NUMBER** into a single `normal_phone`

- Retains static fields like `FIRST_NAME`, `ZIP`, and `GENDER`

- `PERSONAL_EMAILS` contains multiple emails in one string

- Formatting is inconsistent (e.g., `email1, email2`, or `"email1" , email2`)

#### ‚ùì Why is my sync stuck in progress or syncing very slowly?

This typically happens when your query runs too frequently, such as every few minutes, which can **overload the system**.

üëâ **Recommendation:** Set your queries to run **every 1‚Äì3 days**. Since the data refreshes only once per day, running the query more frequently offers no additional benefit and may slow down processing for other syncs.

This occurs when the **primary key** used in your model is **not included in your sync configuration**.

‚úÖ Make sure the **primary key** (e.g., `user_id`, `email`) is present both in your model and in the sync settings. This ensures the system can properly identify and track each record, preventing unnecessary skips.

#### ‚ùì Why does the sync show 0 rows processed even though it‚Äôs active?

This often means you‚Äôre using a **cursor field that doesn‚Äôt exist** in your data source.

üìå For example, if your sync is configured to track changes using a field like `updated_at`, but that field is missing or empty, no records will be found or processed.

‚û°Ô∏è **Solution:** Double-check that the cursor field:

- Is consistently populated with valid values (e.g., timestamps)

#### ‚ùì Why isn't my Facebook destination syncing properly?

This is typically caused by using the **wrong Audience ID**.

‚úÖ **Do use** the **Audience ID from your Facebook Ad Account** (found in Ads Manager).

- ‚úÖ Correct: `23842999999930541` (from Facebook Ads Manager)

- ‚ùå Incorrect: `aud_32903_audience_lab_sync` (from AudienceLab platform)

#### ‚ùì Why can‚Äôt I map fields when syncing to Google Sheets?

This happens when the Google Sheet is missing essential structure.

- Make sure the first row contains **headers** (e.g., `email`, `name`, `timestamp`)

- Include at least **one row of sample data** below the headers

üõ† Without this structure, the system cannot detect or display the available fields for mapping.

#### ‚ùì Why are only 1,000‚Äì10,000 rows syncing while thousands are skipped?

The sync only sends **net new records** each time it runs.

‚û°Ô∏è The sync will process and send only those 1,500 new records

SuperPixel v3 resolves users into persistent, portable profiles using a **multi-layered deterministic system**. Here‚Äôs how we do it:

Resolution starts with a `cookie_id` or `maid_id`. The user must authenticate for us to hash their email (HEM = Hashed Email).

- Captured email must match session IP & user agent

```json
json
{
  "cookie_id": "a1f2c3d4e5",
  "event": "authentication",
  "email_raw": "user@example.com",
  "hem": "4d186321c1a7f0f354b297e8914ab240",
  "ip": "192.168.0.1",
  "user_agent": "Mozilla/5.0"
}
```

> We never use IP-to-HEM guessing. IPs are used only for validation.

Each HEM is deterministically linked to a `uid`. If a user logs in across multiple browsers or devices, we group them under one `uid` when **2 or more identifiers match**:

```json
json

  "uid": "u_01HEABCX789TZY",

  "cookie_ids": ["a1f2c3d4e5", "b9g7k2m3t1"],
  "maids": ["aabbccddeeffgghh"]
}
```

`uid` is mapped to a `profile_id` where all behavioral and identity signals are consolidated:

### Understanding "Lenses" for Machine Learning Categorization
<a id="custom-models-understanding-lenses-for-machine-learning-categorization"></a>

**Source:** https://docs.audiencelab.io/custom-models-/4z72nzTBAcQ8nqMRzSvb33/understanding-lenses-/4z72nzTBAcCgR5kCsmx8Pg

Lenses are structured semantic perspectives that help categorize a topic based on _how the user or customer sees it_ when they land on a webpage, search for a solution, or consume content.

LLMs aren‚Äôt just decoding topics ‚Äî they‚Äôre interpreting **user intent**. So we need to match descriptions to the **perspective a user has** when they arrive at a certain page.

- A user searching for "Apollo.io" is likely trying to learn what the company does (Brand lens)

- A user searching for "cold email deliverability" is trying to solve a problem (Solution lens)

- A user searching for "InboxRamp" is researching a product (Product lens)

By writing with the correct lens, we help the model:

- Avoid noise from comparison pages, forums, or irrelevant media mentions

|     |     |     |     |     |
| --- | --- | --- | --- | --- |
| **Lens** | **Focus** | **Cold Email Software** | **Sales & Marketing CRM** | **Conversational AI Tools** |
| **Brand** | Company or named entity | Instantly.ai | Salesforce | Drift |
| **Product** | Specific product or feature | Smart Inbox | HubSpot Sales Starter | Intercom Inbox |
| **Solution** | The outcome the user wants | Cold email deliverability | Revenue pipeline visibility | 24/7 automated support |
| **Function** | What the tool actually does | Domain warm-up rotation | Lead scoring & CAC payback tracking | Natural language intent detection |
| **Service** | How the solution is delivered | Email outreach as-a-service | CRM onboarding and consulting | Bot-building services |

##### üîß Examples of Header Mappings (Including User Data Fields)

If you're sending metadata like a user's name, email, or phone number, and the receiving endpoint expects plain field names, you can name the headers without the `x-` prefix ‚Äî such as `name`, `email`, or `phone_number`.

#### ‚ùì Why is my sync stuck in progress or syncing very slowly?

This typically happens when your query runs too frequently, such as every few minutes, which can **overload the system**.

üëâ **Recommendation:** Set your queries to run **every 1‚Äì3 days**. Since the data refreshes only once per day, running the query more frequently offers no additional benefit and may slow down processing for other syncs.

This occurs when the **primary key** used in your model is **not included in your sync configuration**.

‚úÖ Make sure the **primary key** (e.g., `user_id`, `email`) is present both in your model and in the sync settings. This ensures the system can properly identify and track each record, preventing unnecessary skips.

#### ‚ùì Why does the sync show 0 rows processed even though it‚Äôs active?

This often means you‚Äôre using a **cursor field that doesn‚Äôt exist** in your data source.

üìå For example, if your sync is configured to track changes using a field like `updated_at`, but that field is missing or empty, no records will be found or processed.

‚û°Ô∏è **Solution:** Double-check that the cursor field:

- Is consistently populated with valid values (e.g., timestamps)

#### ‚ùì Why isn't my Facebook destination syncing properly?

This is typically caused by using the **wrong Audience ID**.

‚úÖ **Do use** the **Audience ID from your Facebook Ad Account** (found in Ads Manager).

- ‚ùå Incorrect: `aud_32903_audience_lab_sync` (from AudienceLab platform)

#### ‚ùì Why can‚Äôt I map fields when syncing to Google Sheets?

This happens when the Google Sheet is missing essential structure.

- Make sure the first row contains **headers** (e.g., `email`, `name`, `timestamp`)

- Include at least **one row of sample data** below the headers

üõ† Without this structure, the system cannot detect or display the available fields for mapping.

### üîÅ Syncs Overview ‚Äì Understanding Data Syncs in AudienceSync

**Syncs** define how your data is **transferred and mapped** from its source to its destination within **AudienceSync**.

- A **destination** (where the data should be delivered).

AudienceSync supports **two sync types**, each serving different use cases:

- **What it does:** Replaces _all_ the data in the destination with the data from the source.

- **When to use it:** When you want to completely overwrite the destination with the latest version from your source.

- **What it does:** Syncs only the data that has **changed since the last sync**.

- **When to use it:** Ideal for keeping your destination up to date with **only new or updated records** without overwriting existing data.

In **AudienceSync**, **streams** refer to the **destination APIs or objects** you are syncing data to.

Each stream represents a different target object in the destination platform where specific data types are synced.

---

### üîÅ Incremental Syncs with Cursor Field ‚Äì Minimizing Data Transfer in AudienceSync

AudienceSync supports an optimized **Incremental Sync** mode using a **Cursor Field**. This allows you to transfer **only new or updated records** by leveraging a field that tracks changes over time‚Äîresulting in **faster syncs** and **less data duplication**.

> üîç This feature is only available for sources that support cursor fields (currently only date fields are supported).

A **Cursor Field** acts as a **timestamp marker** to track what data has changed since the last sync.

- Ensures only **modified or added records** are synced.

- Enables the system to **resume exactly where it left off**.

- Optimizes performance and reduces API strain or data volume.

> ‚ö†Ô∏è Do not edit or remove a cursor field after setup‚Äîit may cause data loss or disrupt sync integrity.

#### ‚öñÔ∏è Handling Ambiguity & Inclusive Cursors

If the cursor field is too coarse (e.g., tracks **only dates**), distinguishing between **old and new records** within the same day may be difficult.

- If a record changes **without updating the cursor field**, it **will not be picked up** in incremental syncs.

- **Editing or deleting** a cursor field **breaks the tracking system**, leading to **potential data loss or inconsistencies**.

> üß∑ To maintain sync reliability: Never change or remove the cursor field after configuration.

### üîÅ Full Refresh Syncs ‚Äì How They Work in AudienceSync

**Full Refresh** is a sync mode in **AudienceSync** used when you want to **completely replace existing data** in the destination with new data from the source.

This method retrieves **all available data**, regardless of what was previously synced, ensuring the destination always holds a **full, current snapshot** of the source.

- You want to **overwrite** all data in the destination.

- You need a **clean slate** with the latest source data.

- It‚Äôs okay to **lose old records** that are no longer present in the source.

- The system **deletes existing rows** in the destination table.

- It then inserts the **entire dataset** from the source.

Let‚Äôs say the destination contains a `Users` table:

#### üì¶ Template 1: **Explode Values Template**

##### ‚úÖ Copy & Paste Version (for AudienceSync)

```sql
SELECT
  -- Combined Email Field
  COALESCE(
    NULLIF(TRIM(split_sha256[idx]), ''),
    NULLIF(TRIM(split_sha256_business_email[idx]), '')
  ) AS normal_email,

-- Combined Phone Field
  COALESCE(
    NULLIF(TRIM(split_mobile_phone[idx]), ''),
    NULLIF(TRIM(split_personal_phone[idx]), ''),
    NULLIF(TRIM(split_direct_number[idx]), '')
  ) AS normal_phone,

-- Static fields (not exploded)
  FIRST_NAME,
  LAST_NAME,
  PERSONAL_CITY,
  PERSONAL_STATE,
  PERSONAL_ZIP,
  GENDER
FROM (
  SELECT
    -- Splitting multi-value fields
    split(SHA256_PERSONAL_EMAIL, ',') AS split_sha256,
    split(PERSONAL_EMAILS, ',') AS split_emails,
    split(DIRECT_NUMBER, ',') AS split_direct_number,
    split(MOBILE_PHONE, ',') AS split_mobile_phone,
    split(PERSONAL_PHONE, ',') AS split_personal_phone,
    split(SHA256_BUSINESS_EMAIL, ',') AS split_sha256_business_email,

-- Static fields passed through
    FIRST_NAME,
    LAST_NAME,
    PERSONAL_CITY,
    PERSONAL_STATE,
    PERSONAL_ZIP,
    GENDER,

-- Index list for row explosion
    range(
      0,
      GREATEST(
        array_length(split(SHA256_PERSONAL_EMAIL, ',')),
        array_length(split(PERSONAL_EMAILS, ',')),
        array_length(split(DIRECT_NUMBER, ',')),
        array_length(split(MOBILE_PHONE, ',')),
        array_length(split(PERSONAL_PHONE, ',')),
        array_length(split(SHA256_BUSINESS_EMAIL, ','))
      )
    ) AS idx_list
  FROM all_sources
) t,
UNNEST(idx_list) AS idx_struct(idx)
WHERE (
  (split_sha256[idx] IS NOT NULL AND split_sha256[idx] <> '')
  OR (split_sha256_business_email[idx] IS NOT NULL AND split_sha256_business_email[idx] <> '')
)
AND (
  (split_mobile_phone[idx] IS NOT NULL AND split_mobile_phone[idx] <> '')
  OR (split_personal_phone[idx] IS NOT NULL AND split_personal_phone[idx] <> '')
  OR (split_direct_number[idx] IS NOT NULL AND split_direct_number[idx] <> '')
);
```

- Combines **SHA256\_PERSONAL\_EMAIL** and **SHA256\_BUSINESS\_EMAIL** into a single column called `normal_email`

- Merges **MOBILE\_PHONE**, **PERSONAL\_PHONE**, and **DIRECT\_NUMBER** into a single `normal_phone`

- Retains static fields like `FIRST_NAME`, `ZIP`, and `GENDER`

- Ensures one clean value per row ‚Äî exactly what Facebook, TikTok, and Google expect

### üì¶ Template 4: **First Email from Raw Text** (AudienceSync)

Extract the **first email address** from a messy, comma-separated list in the `PERSONAL_EMAILS` field ‚Äî even when formatting is inconsistent (spaces, no quotes, weird separators, etc.).

This version is fast, reliable, and does **not require splitting arrays** or cleaning input manually.

- `PERSONAL_EMAILS` contains multiple emails in one string

- Formatting is inconsistent (e.g., `email1, email2`, or `"email1" , email2`)

```plain
srmatthews1967@gmail.com, marhodes34@gmail.com, sharon.matthews@att.net
```

```plain
   first@example.com , second@example.com  , third@example.com
```

#### ‚úÖ Copy & Paste Version (for AudienceSync)

```sql
SELECT
  FIRST_NAME,
  LAST_NAME,
  PERSONAL_ADDRESS,
  PERSONAL_CITY,
  PERSONAL_STATE,
  PERSONAL_ZIP,

-- Extract the first valid email pattern
  REGEXP_EXTRACT(PERSONAL_EMAILS, '[^, ]+@[^, ]+') AS first_personal_email,

SKIPTRACE_WIRELESS_NUMBERS,
  SKIPTRACE_DNC
FROM all_sources
WHERE SKIPTRACE_WIRELESS_NUMBERS IS NOT NULL
  AND SKIPTRACE_WIRELESS_NUMBERS <> ''
  AND PERSONAL_EMAILS IS NOT NULL
  AND PERSONAL_EMAILS <> '';
```

- `REGEXP_EXTRACT(..., '[^, ]+@[^, ]+')` finds the first **valid-looking email** ‚Äî skipping spaces and commas

- Works **even if**`PERSONAL_EMAILS` is sloppy, inconsistent, or has mixed delimiters

- üîÑ Want to get the **second** email instead? Use `regexp_extract_all(...)[1]`

- üßº Want to force lowercase? Wrap with `LOWER(...)`

- üí° Want to use this on another field? Just swap `PERSONAL_EMAILS` with any column

> Even slight topic drift introduces massive noise.

If your topic description includes multiple entities, vague terms, or cross-cutting comparisons, you risk training the model on irrelevant or low-signal content.

- **One topic** (e.g., a brand, feature, or outcome)

- **One lens** (e.g., Brand, Function, Service, etc.)

- **One clear paragraph** (no keywords, comparisons, or fluff)

### Understanding "Lenses" for Machine Learning Categorization
<a id="custom-models-understanding-lenses-for-machine-learning-categorization"></a>

**Source:** https://docs.audiencelab.io/custom-models-/4z72nzTBAcQ8nqMRzSvb33/understanding-lenses-/4z72nzTBAcCgR5kCsmx8Pg

Lenses are structured semantic perspectives that help categorize a topic based on _how the user or customer sees it_ when they land on a webpage, search for a solution, or consume content.

LLMs aren‚Äôt just decoding topics ‚Äî they‚Äôre interpreting **user intent**. So we need to match descriptions to the **perspective a user has** when they arrive at a certain page.

- A user searching for "Apollo.io" is likely trying to learn what the company does (Brand lens)

- A user searching for "cold email deliverability" is trying to solve a problem (Solution lens)

- A user searching for "InboxRamp" is researching a product (Product lens)

By writing with the correct lens, we help the model:

- Avoid noise from comparison pages, forums, or irrelevant media mentions

|     |     |     |     |     |
| --- | --- | --- | --- | --- |
| **Lens** | **Focus** | **Cold Email Software** | **Sales & Marketing CRM** | **Conversational AI Tools** |
| **Brand** | Company or named entity | Instantly.ai | Salesforce | Drift |
| **Product** | Specific product or feature | Smart Inbox | HubSpot Sales Starter | Intercom Inbox |
| **Solution** | The outcome the user wants | Cold email deliverability | Revenue pipeline visibility | 24/7 automated support |
| **Function** | What the tool actually does | Domain warm-up rotation | Lead scoring & CAC payback tracking | Natural language intent detection |
| **Service** | How the solution is delivered | Email outreach as-a-service | CRM onboarding and consulting | Bot-building services |
| **Event** | Where users learn or connect | Cold Email Bootcamp | Dreamforce | Conversational AI Summit |

> ‚úÖ Choose ONE lens per topic to minimize ambiguity and maximize precision in model behavior.
